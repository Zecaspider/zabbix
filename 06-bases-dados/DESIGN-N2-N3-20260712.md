## 1. O que o Zabbix realmente tem hoje (26 hosts de Produção)

Foco: `camada=base de dados` + `ambiente≈Produção` (a tag tem casing
inconsistente: "Produção"/"Producao"/"Produção" — o filtro tem de ser
case-insensitive `startsWith("produ")`). **26 hosts** (não 23 — a contagem
anterior usava match exacto e perdia os de casing diferente).

### 1.1 Três fontes de dados reais, por ordem de riqueza

| Fonte | Template Zabbix | Cobre | O que dá |
|---|---|---|---|
| **ODBC completo** | `BPC MSSQL by ODBC` (78 items) | **1 host** (`VS8000413`) — só onde a credencial `zbx_monitor` já existe | Tudo: 27 bases com nome/estado/recovery/tamanho/último backup, PLE, cache hit, batch req/s, deadlocks, jobs falhados, ligações, bloqueados |
| **Perfmon+WMI local** | `BPC MSSQL Perfmon sem Credencial` | **3 hosts** (`VS8000435`, `VS8000601`, `VS8000790`) | Contadores de desempenho MSSQL (PLE, cache, ligações, deadlocks/s) **+** tabela de bases por WMI (nome, tamanho dados/log, % log usado) — sem SQL |
| **Serviço (Windows)** | `Windows by Zabbix agent active` (service discovery) | **todos os Windows** | `service.info["MSSQLSERVER",state]` e afins descobertos automaticamente — estado running/parado do motor, sem template extra |
| **SO base** | `Windows by Zabbix agent active` | **todos** | CPU %, RAM % (`vm.memory.util`), disco por volume (`vfs.fs.size[C:,pused]`), uptime, swap |

### 1.2 Porque não há dados SQL na maioria (o "motivo" pedido)

Repartição do erro do item ODBC `db.odbc.select[mssql.business.master,…]`
nos 26 hosts de produção:

| # | Erro ODBC | Significado | Acção |
|---|---|---|---|
| **16** | `28000` / `18456 Login failed` | Porta 1433 aberta, motor responde, mas **falta a credencial `zbx_monitor`** | Pedir criação da credencial à infra |
| **7** | `HYT00` timeout | **Instância nomeada** (porta dinâmica via SQL Browser) — o template ODBC não liga estruturalmente | Perfmon local (já feito em 3) ou DSN por instância |
| **1** | (funciona) | `VS8000413` — credencial já existe | — (é o "farol" do tier completo) |
| **1** | `SQL_ERROR` genérico | `SV9000401` — host **em baixo** (uptime 0, CPU 0) | Recuperar host |
| **1** | sem template ODBC | `VS8000229` (WSUS) — usa Windows Internal Database, não é SQL Server real | Excluir do âmbito ou marcar N/A |

**Mensagem central para o dashboard**: "sem credencial" ≠ "sem
monitorização". 26/26 têm sinal de SO; ~9/26 têm sinal do motor (serviço ou
perfmon); 1/26 tem o conjunto SQL completo. O dashboard mostra exactamente
onde está cada um e sobe o host de tier sozinho quando a credencial aparecer.

## 2. KPIs de monitoria de BD (pesquisa) — o que vale a pena mostrar

Consolidação das métricas mais usadas em monitoria de SGBD (SQL Server em
primeiro lugar, aplicável a Oracle com nomes equivalentes), cruzada com o
que o Zabbix aqui expõe.

### 2.1 Contexto operacional (inventário — o "quanto existe")

Pedido explícito. Responde a "o que estou a monitorizar" antes de "como está".

| KPI de contexto | Fonte disponível | Notas |
|---|---|---|
| **Nº de instâncias** | serviços `MSSQL*`/`Oracle*` descobertos | 1 host pode ter várias (default + nomeadas) |
| **Nº de bases de dados** | ODBC `db_online` **ou** contagem WMI | 27 em VS8000413; 13 em FENIX |
| **Tamanho total (dados+log)** | ODBC `total_data_mb`/`total_log_mb`; WMI soma | VS8000413 = ~1.8 TB dados + ~900 GB log |
| **Motor** | serviço descoberto (MSSQL/Oracle) + tag `motor` (a criar) | hoje inferido; tag ainda não existe |
| **Versão / edição** | ODBC (quando há credencial) | só onde ODBC liga |
| **Recovery model por base** | ODBC `databases[].recovery_model` | FULL vs SIMPLE — indica quem gera log |

### 2.2 Saúde / performance (o "como está")

Ordenados por consenso de utilidade operacional em NOC:

| KPI | Porque importa | Fonte | Threshold usual |
|---|---|---|---|
| **Estado do motor** (up/down) | 1ª pergunta sempre | serviço/perfmon/ODBC | parado = crítico |
| **Bases suspect/offline** | corrupção/indisponibilidade | ODBC `db_suspect` | >0 = crítico |
| **PLE** (Page Life Expectancy) | pressão de memória do buffer | ODBC/perfmon | <300 s aviso, <100 crít |
| **Buffer cache hit ratio** | eficiência de cache | ODBC/perfmon | <90% aviso |
| **% log usado por base** | risco de disco cheio / log full | WMI/ODBC | >90% crítico |
| **Ligações activas** | carga / saturação de sessões | ODBC/perfmon | contexto |
| **Bloqueados / deadlocks/s** | contenção | ODBC/perfmon | >0 investigar |
| **Batch requests/s** | throughput | ODBC/perfmon | baseline |
| **Jobs SQL Agent falhados hoje** | backups/ETL falhados | ODBC | >0 investigar |
| **Backups nas últimas 24h** | RPO / recuperabilidade | ODBC `backups_24h` + `last_full_backup` | 0 = risco |
| **Uso de disco do volume de dados/log** | causa nº1 de paragem de BD | SO `vfs.fs.size` | >85% aviso |
| **CPU / RAM do host** | saturação de recursos | SO Windows | >90% aviso |

Princípio de desenho: **nunca mostrar um número que a fonte do host não
suporta** — cada tier expõe o subconjunto que consegue e diz o que falta.

## 3. Bug encontrado no N2 atual (a corrigir na reconstrução)

`l2-tabela-hosts.js` §157: o ramo "serviço" faz
`anyDown = states.some(s => !s.running)` sobre **todos** os `service.info`
do host. Mas o service-discovery do Windows traz serviços auxiliares que
estão legitimamente parados (`SQLBrowser`, `MSSQLServerOLAPService`,
`MSSQLFDLauncher`, `OracleServiceRMAN*` = backup). Resultado: um host com o
**motor a correr** mas um auxiliar parado é marcado **crítico "motor
parado"** — falso positivo.

**Correcção**: distinguir o *serviço primário do motor* dos auxiliares.
- Primário MSSQL: `MSSQLSERVER` (default) ou `MSSQL$<instância>` (nomeada)
- Primário Oracle: `OracleService<SID>` **excepto** `*RMAN*`/`*Backup*`
- Auxiliares (ignorar na saúde, mostrar só como contexto): FDLauncher,
  Launchpad, OLAP, SQLBrowser, RMAN, Agent (o Agent parado é aviso, não crít)

## 4. Duas propostas para o N2

Mesmos dados, duas filosofias de apresentação. Ambas focam **produção**.

### Proposta A — Business Text (evolução do actual) · `bd-n2`

Superfície NOC única e composta, 100% controlada. Blocos:

```
┌ CONTEXTO OPERACIONAL ────────────────────────────────┐
│ 26 hosts · N instâncias · N bases · X TB · motores    │  ← NOVO
│ [MSSQL 24] [Oracle 1] [outros 1]                      │
├ COBERTURA DE MONITORIZAÇÃO ──────────────────────────┤
│ ● ODBC 1  ◐ Perfmon 3  ◑ Serviço 5  ○ Sem sinal N     │  (existente, afinado)
├ ATENÇÃO IMEDIATA ────────────────────────────────────┤
│ achados gerados: Oracle parado, log 80%, 275 jobs...  │  (existente, +motor)
├ SAÚDE AGREGADA (só hosts com dados reais) ───────────┤
│ PLE mín · ligações · deadlocks · jobs · backups 24h   │  (existente)
├ HOSTS (tabela, ordenada por severidade) ─────────────┤
│ Estado│Host│Sistema│Motor│Bases│Tamanho│Fonte│→N3     │  (+ Bases, Tamanho)
└──────────────────────────────────────────────────────┘
```

**Prós**: lógica de tier + parsing JSON + agregação WMI só é possível em
código; achados operacionais em linguagem natural; estética NOC coerente;
uma só superfície. **Contras**: tudo é código custom (manutenção).

### Proposta B — Nativo Grafana · `bd-n2-nativo` (UID novo, coexiste)

Mesmos dados por painéis nativos:
- **Stat row** (KPIs): nº hosts, nº bases, tamanho total, hosts sem
  credencial, jobs falhados — via `ds/query` MySQL/Zabbix
- **Bar gauge / pie** de cobertura por tier
- **Tabela nativa** de hosts (transform de `item.get` ou MySQL) com sort/filtro
  nativos do Grafana
- **Painel nativo de problemas Zabbix** (`alexanderzobnin-zabbix-triggers-panel`)
  filtrado ao grupo — triggers reais de BD

**Prós**: menos código, sort/filtro/paginação nativos, painel de problemas
oficial, séries temporais grátis. **Contras**: os dados-chave deste domínio
são **JSON dentro de 1 item** (o master ODBC) e **agregação por base (WMI)** —
painéis nativos não fazem parsing de JSON aninhado nem a lógica de tier; a
cobertura por tier tem de ser pré-computada. O nativo brilha nas **métricas
de SO** (CPU/RAM/disco em série temporal) e no **painel de problemas**, não
na inteligência de tier.

### Recomendação

**N2 = Proposta A (BT)** como principal — a inteligência de tier/JSON/WMI é o
valor deste domínio e não existe em nativo. Construir a **Proposta B em
paralelo** (UID `bd-n2-nativo`) para comparação lado a lado no Grafana, com
o painel de problemas nativo como peça que **vale a pena adoptar mesmo na A**.
**N3 = híbrido**: cabeçalho/saúde/bases em BT (tier), séries de SO
(CPU/RAM/disco) e problemas em **nativo** — o melhor de cada.

## 5. N3 · Instância (por host, `var-hostid`)

Standalone (reage a `var-hostid` sem estado residual — padrão de Servidores
Virtuais). Variável via MySQL `MYSQL - INFRA ZABBIX` (tag `camada`, sem grupo
Zabbix dedicado). Secções (mockup CLAUDE.md §5, agora com dados confirmados):

1. **HEADER** — nome, tags (sistema/dept/ambiente), badge de Fonte/tier
2. **CONTEXTO** — motor, versão (se ODBC), nº instâncias, nº bases, tamanho
3. **SAÚDE (tiered)** — muda com o tier: ODBC=completo; perfmon=contadores;
   serviço=só estado; sem sinal=próximo passo
4. **BASES DE DADOS** — tabela por base (ODBC `databases[]` ou WMI), ordenada
   por % log usado; colunas: nome, estado, recovery, dados, log, %log, último
   backup — apanha PRISIGEP/DWDiagnostics cedo
5. **RECURSOS DO SO** (nativo) — CPU %, RAM %, disco por volume (série temporal)
6. **PROBLEMAS** (nativo Zabbix triggers panel)
7. **FICHA** — SO/IP/cluster/host físico (sempre presente, independente do tier)

Hosts de teste: `VS8000413` (ODBC completo), `VS8000601`/`VS8000790`
(perfmon+WMI), `VS8000430` (só serviço), `VS8000759` (Oracle parado),
`SV9000401` (sem sinal/host down).
